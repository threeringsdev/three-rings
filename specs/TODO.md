## Decisions log

- 2026-07-16: **collection-api implemented ŌĆö the full endpoint surface, both backends, verified on dev.** Built as six vertical slices, each proven end-to-end against the Neon dev branch before commit: tree CRUD + lazy inbox; holdings/desires writes + batch; CollectionView reads; moves (single/batch, undo, teardown, suggested-destinations); global reads (all-cards/needs/shopping); catalog (detail/summary/search shell). Every operation is a `CatalogStore`/`CollectionStore` trait method with a `shared/` DTO, implemented once by `HostedBackend` (sqlx) and once by `NativeBackend` (the HTTP client of explicit hosted JSON routes; paths in one shared module so they can't drift). Decisions settled in implementation: **endpoint surface = the explicit routes** (no per-op Leptos server-fns ŌĆö those are a per-screen UI adapter, ride with the UI tasks); **`CardRow` grain = (printing, board)** with owned = global per-oracle, rollup = descendants per printing, desired = per (oracle, board); **moves are board-agnostic** (mainboard ŌĆö board re-labels stay card-tagging's); **keyset pagination** on the large reads (opaque cursor + limit+1), **needs/shopping unpaginated** (bounded; follow-up filed); **catalog reads auth opportunistically** (HeaderMapŌåÆuser_id, else anonymous); **ownership guard** validates the target collection is owned beneath the `holdings`/`desires` RLS (which only gates on their own `user_id`).
